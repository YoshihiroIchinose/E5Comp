# Purview での日本語のマッチング処理について
Purview でカスタムの機密情報の種類でキーワード リストの文字列の一致を用いた場合のマッチングの仕組みについて、各種テストから判明した動作についてまとめておきます。
1. カスタムの機密情報に定義されたキーワード リストを Unicode で定義されている FormKC を用いて正規化する
2. 検証対象となる本文中の文字列を Unicode で定義されている FormKC を用いて正規化する
3. 2 の文字列をさらにワード ブレークの処理で単語ごとに半角スペースで区切った文字列も用意する
4. 1 のキーワードが、2 や 3 のいずれかの文字列に含まれていないマッチングを行う
5. 2 や 3 のいずれかでマッチした文字列があれば、ワード ブレークする前の文字列をマッチング箇所とする

## FormKC の補足
- FormKC 正規化は、文字としての表記ゆれを解消するもので、全角英数記号は、半角英数記号に変換され、半角カナは全角カナに変換されます。   
例 "東京２０２０" -> "東京2020"

- 半角カナの場合、濁音、半濁音の記号自体が個別の文字となっていますが、FormKC の正規化では、直前の半角カナと濁音・半濁音は合成され、
一つの全角カナ文字に変換されます。   
例 "ﾊﾟｰﾋﾞｭｰ" -> "パービュー"

- 本文および、キーワード リスト側で双方同じ FormKC の正規化処理が行われるため、英数記号のキーワードは半角文字で登録しておくことで、
　本文中の全角の英数記号も合わせて検知できる仕組みとなっています。

- PowerShell からも以下のようなコマンドで、FormKC の変換結果を簡単に確認できます。
````
"ﾏｲｸﾛｿﾌﾄ ﾊﾟｰﾋﾞｭｰ".Normalize([System.Text.NormalizationForm]::FormKC)
````

## ワード ブレーク処理の補足
- ワード ブレークの動作を手元の環境で検証することはできませんが、ワード ブレークの処理により、全角と半角が隣り合う箇所では、
半角カナを挿入するような処理も行われます。    
例 "東京2020" -> "東京 2020"   
   "東京都知事2020" -> "東京 都 知事 2020"

## 実際のテスト ケース
カスタムの機密情報のキーワード リストを使った文字列の一致の以下のテスト ケースから、本文中と全く同じ文字列の場合か、ワード ブレークされた場合と同じ分割の文字列の場合でのみヒットすることが分かります。(テスト ケース A,C)
また同時に、キーワード リストの方では、ワード ブレークの処理がマッチングに際して行われないことが分ります。(テスト ケース D,E)

|Case| キーワード | 本文 | マッチ結果|
|--------|--------|--------|--------|
|A|東京都知事2020|東京都知事2020| 〇 |
|B|東京都知事 2020|東京都知事2020| × |
|C|東京 都 知事 2020|東京都知事2020| 〇   |
|D|東京 都 知事2020|東京都知事2020| × |
|E|東京都 知事 2020|東京都知事2020| × |

# マッチした箇所として検出される文字列について
カスタムの機密情報の種類で指定したキーワード リストのキーワードが本文をワードブレークした後の 1 つの単語に包含される場合は、一致した文字列だけではなく、その単語も検出されます。
1. カスタムの機密情報に定義されたキーワード リストを Unicode で定義されている FormKC を用いて正規化する
2. 検証対象となる本文中の文字列を Unicode で定義されている FormKC を用いて正規化する
3. 2 の文字列をさらにワード ブレークの処理で単語ごとに半角スペースで区切った文字列も用意する
4. 1 のキーワードが、2 や 3 のいずれかの文字列に含まれていないマッチングを行う
5. 2 や 3 のいずれかでマッチした文字列があれば、ワード ブレークする前の文字列をマッチング箇所とする
6. 基本的にキーワードで指定した文字列を本文が含む場合、キーワードの文字列は検出される (テスト ケース A-F)
7.さらに、キーワードに含まれる文字列が本文をワード ブレークした一つの単語に包含される場合一致した文字列の箇所だけではなく、その単語そのものも検知される (テスト ケース A,C,D,E,F)
6.ただし、キーワードが、本文をワードブレークした単語に先頭から部分一致した場合には、6 の一致した文字列での検知は上書きされ、7 の単語の検知のみが表示される (テスト ケース G-I)
 
## 実際のテスト ケース
ワード ブレークの処理では、"経済産業省"は一単語として扱われ、"経済産業"は、"経済" + "産業"の 2 単語に分割されます。

|Case|キーワード|本文|マッチ結果|
|--------|--------|--------|--------|
|A|済産|経済産業省|済産, 経済産業省|
|B|済産|経済産業|済産|
|C|済|経済産業省|済, 経済産業省|
|D|済|経済産業|済, 経済|
|E|業|経済産業省|業, 経済産業省|
|F|業|経済産業|業, 産業|
|G|経|経済産業省|経済産業省|
|H|経|経済産業|経済|
|I|経済|経済産業省|経済産業省|
|J|経済|経済産業|経済|

# キーワード辞書での文字列の一致について
2025 年末から 2026 年にかけて提供されている新機能として、キーワード辞書において、キーワード リストと同様に、単語の一致だけではなく、
文字列の一致が選べるようになっています。この機能が出る前は、キーワード辞書においては、文字列の一致や単語の一致を選択することができず、
動作としては、単語の一致のみの動作となっていました。この新機能でキーワード辞書においても、単語の一致が利用できるようになりましたが、
キーワード リストの文字列の一致とは、少し動作が異なるものとなっています。おおよその動作は以下となっています。

1. カスタムの機密情報に定義されたキーワード辞書のキーワードを Unicode で定義されている FormKC を用いて正規化する
2. 1 のキーワードをさらに、ワード ブレークの処理で単語ごとに半角スペースで区切った文字列を用意する   
   例: "新宿区" -> "新宿 区"
3. 検証対象となる本文中の文字列を Unicode で定義されている FormKC を用いて正規化する
4. 3 の文字列をさらにワード ブレークの処理で単語ごとに半角スペースで区切った文字列を用意する
   例: "西新宿区画整理" -> "西 新宿 区画 整理"
6. 2 キーワード辞書に含まれる単語の並びで、先頭の単語は後方一致も許容し、最後の単語は前方一致も許容する形でマッチングを行う   
   例: "新宿 区" -> "\*新宿 区\*"
7. 6 の後方一致、前方一致も許容した上で、本文中の分割された単語単位でマッチングを行い、見つかった単語を連携したものをマッチした文字列とする
   例: "\*新宿 区\*" で "新宿 区画"にマッチし、本文中の"新宿区画"をマッチした文字列とする

## 実際のテスト ケース
キーワード辞書での文字列の一致は、引き続きワード ブレークにも依存した仕組みとなっていて、ケース D のようにキーワード側が単語が 2 つ、本文側が 単語 1 つのように、単語の数が一致しない場合、一見して文字列の一致があっても、マッチしません。ケース E のように、本文側がワード ブレークされない"経済産業省"ではなく、"経済"と"産業"にワード ブレークされる"経済産業"の場合、"済産"はヒットします。ケース J のように、本文の文脈によりワード ブレークの位置が異なる場合("坂 井田"と"坂井 田")にも一致しません。
